# Patent Writing Studio Role Auth 面试讲解稿

## 1. 一句话介绍

这是一个面向专利撰写场景的本地优先 LLM 工作台，把专利写作前期常见的资料整理、写作风格沉淀、背景资料生成、模板生成和模板搭建做成一条可演示的工作流，并补充了管理员/普通用户两类角色权限。

面试时可以这样开场：

> 我做的是一个专利 LLM 写作辅助工具。它不是简单的聊天框，而是把专利撰写拆成几个更稳定的环节：先整理技术主题和现有技术，再蒸馏写作风格，再生成可编辑模板，最后用模块化搭建器继续调整模板。这样可以减少大模型直接生成专利文本时容易出现的空泛、跑题和结构不稳定问题。

## 2. 项目背景

专利撰写不是普通文案生成，核心难点有三个：

- 需要围绕技术问题、技术方案和技术效果展开，不能写成宣传稿。
- 权利要求、说明书、实施例之间要有结构一致性。
- 不同代理师或公司有自己的写作风格和模板习惯，不能每次从零开始。

所以这个项目的设计思路是：不把 LLM 当成一个万能回答器，而是把它放进专利撰写流程中，让每个页面负责一个明确阶段。

## 3. 用户流程

完整流程可以按 5 步讲：

1. 登录系统

   用户先登录，系统根据角色加载不同权限。管理员可以管理用户和 API 设置，普通用户只能使用写作流程。

2. 风格蒸馏

   输入或选择代理师样本文本，系统提取专利写作风格，例如段落结构、句式长度、实施方式展开方式、权利要求表达习惯。

3. 背景资料

   输入技术主题，系统整理相关论文方向、专利线索、官方检索入口、创新点和技术流程，为后续模板生成提供上下文。

4. 模板工坊

   把技术主题、背景资料和风格画像组合起来，生成一份带留白的专利模板底稿。用户也可以上传已有模板，让系统保持原有结构。

5. 模板搭建器

   用模块化编辑方式搭建 A4 模板页，把标题、段落和内容块组合成更可控的模板，再同步回模板工坊。

## 4. 功能模块怎么讲

### 风格蒸馏

风格蒸馏模块对应 `src/style-distiller.js`。

可讲重点：

- 它不只是总结文本，而是提取可复用的写作偏好。
- 输出结果会影响后续模板生成，使模板更接近指定代理师或指定风格。
- 当大模型不可用时，也有本地规则兜底，保证演示流程可运行。

### 背景资料生成

背景资料模块对应 `src/background-generator.js` 和 `src/source-guides.js`。

可讲重点：

- 这个模块把技术主题拆成关键词、研究方向、论文线索、专利线索和方法步骤。
- 它强调“先组织现有技术，再写底稿”，避免 LLM 直接凭空编写。
- 官方检索入口用于提醒用户回到 CNIPA、USPTO、WIPO 等来源核验，不伪造检索结论。

### 模板生成

模板模块对应 `src/template-engine.js`。

可讲重点：

- 模板生成不是直接输出最终专利，而是输出可编辑的底稿框架。
- 模板中保留待补充位置，便于人工继续补充真实实施例、实验数据和权利要求细节。
- 这样比“一次性生成全文”更适合专利工作，因为专利文本需要审查和逐步收敛。

### LLM 接入

LLM 接入模块对应 `src/llm-client.js` 和 `src/ai-orchestrator.js`。

可讲重点：

- 使用 OpenAI 兼容接口，支持自定义 Base URL、API Key 和模型名。
- `llm-client` 负责通用请求、JSON 提取、错误处理。
- `ai-orchestrator` 负责把风格、背景、模板等业务任务组织成结构化提示词。
- 未配置模型时可以使用本地规则，便于无网络或无密钥环境下演示。

### 用户与权限

用户模块对应 `src/user-store.js`。

可讲重点：

- 系统有管理员和普通用户两种角色。
- 管理员可以管理账号、重置密码、配置 API。
- 普通用户不能看到 API Key 等敏感配置。
- 登录会话和用户工作区都保存在本地 `.local/` 目录，按用户隔离。

## 5. 技术架构

可以这样讲：

> 后端没有使用复杂框架，而是用 Node.js 原生 HTTP 服务实现。这样项目结构比较透明：`server.js` 负责路由和静态文件服务，`src/` 下面按业务能力拆成多个模块，前端页面放在 `public/`。本地数据用文件存储，适合个人工具和面试演示；如果要做生产版，可以把文件存储替换成数据库，把会话和密码策略升级。

架构分层：

- `public/`：页面和浏览器端交互。
- `server.js`：HTTP 服务、API 路由、鉴权入口、静态文件。
- `src/*`：业务模块，包括风格、背景、模板、LLM、用户、工作区。
- `.local/`：本地运行数据，不提交 GitHub。
- `tests/`：模块测试。

## 6. 面试亮点

建议重点讲这 5 个亮点：

1. 不是普通聊天框，而是流程化 LLM 应用。

   项目把专利写作拆成多个环节，降低大模型一次性生成全文的不稳定性。

2. 领域约束明确。

   专利写作强调技术问题、技术方案、技术效果和权利要求支持关系，所以系统用模板和阶段性产物约束输出。

3. 本地优先，演示稳定。

   即使没有 API Key，也能通过本地规则跑通核心流程；有模型配置时再增强生成效果。

4. 角色权限完整。

   区分管理员和普通用户，敏感配置只给管理员看，工作区按用户隔离。

5. 工程上可测试、可扩展。

   核心逻辑拆在 `src/` 里，并配了 `node --test` 测试，后续可以接数据库、接真实专利检索 API、接企业账号体系。

## 7. 可以主动承认的不足

面试时不要把原型吹成生产系统。可以这样说：

> 这个项目目前定位是本地优先的原型和演示系统，不是直接公网部署的生产系统。当前用户和会话使用本地文件保存，适合个人电脑或小范围演示。生产化时我会优先改三件事：第一，把用户和工作区数据迁移到数据库；第二，引入更严格的鉴权、密码策略和密钥管理；第三，接入真实授权的专利/论文数据源，并对 LLM 输出做引用校验和事实核验。

这样讲会更可信。

## 8. 面试常见问题与回答

### Q1：为什么不用一个聊天框直接生成专利？

答：

因为专利文本结构要求很强，直接让大模型生成全文容易出现跑题、重复、权利要求和说明书不一致的问题。我把流程拆成风格、背景、模板和搭建几个阶段，是为了让每一步产物都可检查、可复用，最后生成的模板也保留人工修改空间。

### Q2：LLM 在项目里具体起什么作用？

答：

LLM 主要做三类增强：风格画像增强、背景资料整理增强、模板生成增强。它不是唯一逻辑来源，系统还有本地规则兜底。这样做是为了避免没有 API Key 或模型失败时整个应用不可用。

### Q3：项目怎么保证 API Key 不泄露？

答：

API Key 只允许管理员配置，普通用户看不到。配置保存在本地 `.local/app-settings.json`，这个目录已经加入 `.gitignore`，不会提交到 GitHub。真实生产环境中还应该进一步改成环境变量或密钥管理服务。

### Q4：为什么使用 Node 原生 HTTP，而不是 Express？

答：

这个项目是本地优先原型，我希望依赖尽量少，启动和部署简单，所以用 Node 原生 HTTP 就能满足静态页面、API 路由和 JSON 请求处理。后续如果接口增多，可以平滑迁移到 Express、Fastify 或其他框架。

### Q5：你在这个项目里最想强调的工程能力是什么？

答：

我想强调的是把 LLM 能力产品化，而不是只会调接口。这个项目里我做了流程拆解、模块分层、权限控制、本地数据隔离、LLM 失败兜底和自动化测试。它体现的是一个领域工具从想法到可演示产品的完整闭环。
